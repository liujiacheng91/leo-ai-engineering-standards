## 一、工具链与执行环境

### Item 16 — JGit + Worktree 兼容性标准化处理方案
- **状态**：✅ 已完成
- **优先级**：高
- **来源**：20260511 / 20260512 / 20260513 三个案例均触发，`net.nemerosa.versioning` 插件用 JGit 读 `.git` 时遇到 worktree 文件指针格式，抛 `NoHeadException`，构建配置阶段中止
- **影响**：QA 无法在 worktree 内执行测试 → 编译类问题只能合并后暴露 → 直接驱动 Retry Count 上升（20260512 Retry=2 的第 1 轮根因）
- **建议**：
  1. **短期**：在 `Solution.md` Technical Context 节增加必填项：是否存在 worktree 构建限制？若是，声明"post-merge test plan"（具体命令 + 执行人 + 时机）
  2. **长期**：在 `templates/Solution.md` 中加入 worktree 兼容性检查项；在 `standards/AI_Verification_Standard.md` 中定义"worktree build limitation"豁免条件：若已声明 post-merge test plan，worktree 内无法执行构建不计入 Retry
- **落地文件**：`templates/Solution.md`、`standards/AI_Verification_Standard.md`、`skills/verification/SKILL.md`

### Item 17 — 合并前静态验证 + 合并后测试运行双轨协议
- **状态**：✅ 已完成
- **优先级**：高
- **来源**：20260512 / 20260513 中，构建不可用时 Verify.md 只做了静态 trace；合并后才执行 `gradle test`，但没有标准化协议
- **建议**：在 `skills/verification/SKILL.md` 中明确定义两轨验证协议：

  ```
  轨道 A — 合并前（worktree 内可执行时）：
    1. gradle compileJava / bootJar
    2. gradle test
    3. 静态代码断言 + AC trace
  
  轨道 B — 合并前构建不可用时（声明 worktree limitation 后）：
    1. 静态代码断言 + AC trace（必须）
    2. TA code-review（必须）
    3. Verify.md 中显式声明"post-merge test plan"：命令 + 执行人 + 时机（必须）
    4. 合并后：执行 gradle test，结果回填 Verify.md §补录
  ```
- **落地文件**：`skills/verification/SKILL.md`、`templates/Verify.md`

## 二、Solution.md 质量强化

### Item 18 — Solution.md 必须声明 Technical Constraints
- **状态**：✅ 已完成
- **优先级**：高
- **来源**：20260514 VBO 案例：假设 `String.repeat()` 可用（Java 11 方法，compile target 受限）→ 编译错误；假设 `User.userName` 字段存在 → 字段不存在；均为"未验证直接断言"，触发 Retry

  ```markdown
  |---|---|---|
  | Java 语言级别 / compile target | 读 build.gradle `sourceCompatibility` | |

  | 可用工具方法（JDK / 框架版本相关） | 读 build.gradle 依赖版本 | |
  | Mockito strictness 设置 | 读现有测试类或 build.gradle | |
  | worktree 构建可行性 | 执行 `gradle compileJava` 验证 | |
  ```
- **落地文件**：`templates/Solution.md`、`skills/solution-design/SKILL.md`

## 三、需求分析阶段质量

### Item 19 — RA Draft 质量门控：移交 TA 前自检清单
- **状态**：✅ 已完成
- **优先级**：中
- **来源**：20260511 First Round 案例：TA review 返回 11 个 must-fix 项，触发 RA(draft) → TA(review) → RA(revise) → RA(finalize) 共 3 轮 RA 调用，需求阶段消耗 274,853 tokens（占整案 35%）
- **根因**：RA draft 在不明确的假设点上过早锁定结论，未预先对照 AC 做自检
- **建议**：在 `skills/scenario-analysis/SKILL.md` 中加入 RA 自检清单，移交 TA 前必须通过：

  ```
  RA 移交 TA 自检清单（所有项必须明确，否则不移交）：
  - [ ] 所有业务规则有明确数据来源（表 / 接口 / 文档），无推断
  - [ ] 边界条件和异常路径已明确定义，无"待确认"
  - [ ] AC（验收条件）与需求逐一对应，无遗漏
  - [ ] 已知遗留问题（known issues）明确列出并标注是否在本 case 范围内
  - [ ] 改动范围已列出受影响文件 / 模块，无猜测
  ```

  同时在 AI_Case_Card 中记录：若 TA 返回 > 5 must-fix，标注为"RA 质量偏差"，记录根因。
- **落地文件**：`skills/scenario-analysis/SKILL.md`、`templates/AI_Case_Card.md`

### Item 21 — Saved Hours 计算口径统一
- **状态**：✅ 已完成
- **优先级**：中
- **来源**：4 个案例计算方式不一致：20260511 用"员工累计工作时间"、20260512 用"人工估时 - AI 辅助实际时间"、20260514 估算 25h 但未说明方法
- **建议**：在 `standards/AI_Token_Cost_Standard.md` 中定义统一公式：

  ```
  Saved Hours = Manual Estimate − AI-Assisted Actual Time

  Manual Estimate：人工独立完成同等工作的合理估时（含代码阅读 + 设计 + 实现 + 测试 + 文档）
  AI-Assisted Actual Time：用户实际投入的时间（含 prompt 编写 + review + 决策点等待）

  注意：不包含 AI 的纯机器运行时间（SubAgent 执行的 33 分钟不计入"用户时间"）
  ```
- **落地文件**：`standards/AI_Token_Cost_Standard.md`、`templates/Token_Usage_Report.md`（字段说明）
