## 一、QA 工程化模板缺口

### Item 24 — API 测试模板
- **状态**：✅ 已完成
- **优先级**：高
- **来源**：会议 Section 三.2 — VBO 案例中 API 测试发现了单元测试无法暴露的问题（接口异常、状态流转错误、权限缺失）；Section 五.3 明确建议补充 API 测试模板
- **现状**：`templates/Test.md` 覆盖单元测试、集成测试、边界用例，但无 API 测试的标准化结构（请求构造、认证头、状态码断言、响应体校验、错误码覆盖）
- **建议**：在 `templates/Test.md` 中新增 `## API Test Cases` 节，或创建独立的 `templates/API_Test.md`：

  ```markdown
  ## API Test Cases

  | API | Method | Auth | Request Body | Expected Status | Expected Response | Related AC |
  |---|---|---|---|---|---|---|
  ```

  同时补充：错误码覆盖表、权限矩阵测试表
- **落地文件**：`templates/Test.md` 或 `templates/API_Test.md`、`skills/test-design/SKILL.md`

### Item 25 — 权限与安全测试 Checklist
- **状态**：✅ 已完成
- **优先级**：高
- **来源**：会议 Section 二.3 — VBO 案例覆盖权限控制、状态机流转；Section 五.3 建议建立权限与安全测试 Checklist
- **现状**：`templates/Test.md` 有 "No high-risk security issue" 和 "No sensitive data exposed" 两项 Pass Criteria，但无结构化的权限测试检查项
- **建议**：在 `templates/Test.md` 中新增 `## Permission & Security Checklist` 节：

  ```markdown
  ## Permission & Security Checklist

  | Check Item | Test Method | Result |
  |---|---|---|
  | 未授权用户访问受保护 API | 不带 Token 请求 → 401 | |
  | 低权限用户访问高权限功能 | 用 Reader 角色请求 Admin API → 403 | |
  | SQL 注入测试 | 注入常见 payload → 无异常执行 | |
  | XSS 测试（若有前端） | 注入 script 标签 → 被转义 | |
  | 敏感数据是否脱敏 | 检查日志和响应体 → 无明文密码/身份证等 | |
  ```
- **落地文件**：`templates/Test.md`、`skills/test-design/SKILL.md`

### Item 26 — UAT Ready 检查标准
- **状态**：✅ 已关闭（依赖 SIT 沙箱环境，属基础设施建设范畴；验证不可用场景已由 Track B + Verify.md Caveats 覆盖）
- **优先级**：中
- **来源**：会议 Section 五.3 建议补充 UAT Ready 检查标准；Section 二.3 提到 API 测试更接近真实 UAT
- **现状**：`templates/Verify.md` 有 UAT 行（Pass/Fail/Not Run），但无 UAT 准入的前置检查清单
- **建议**：在 `templates/Verify.md` 或独立模板中定义 UAT Ready 条件：

  ```markdown
  ## UAT Ready Checklist

  - [ ] 所有单元测试通过
  - [ ] API 关键路径测试通过
  - [ ] 权限矩阵测试通过
  - [ ] 状态机流转测试通过
  - [ ] 无 P0/P1 级别未修复缺陷
  - [ ] 测试环境数据已准备
  - [ ] 操作手册或 API 文档已更新
  ```
- **落地文件**：`templates/Verify.md` 或 `templates/UAT_Ready_Checklist.md`

## 二、需求结构化缺口

### Item 27 — BA 需求结构化输入模板
- **状态**：✅ 已完成
- **优先级**：高
- **来源**：会议 Section 二.4 — "BA 文档如能按结构输出，将显著提升 AI 效率"；Section 五.1 — 需求质量仍是关键瓶颈，需推动 BA 需求结构化
- **现状**：`templates/Scenario.md` 面向 AI/开发视角（Business Background / Current Problem / Expected Outcome / AC），缺少面向 BA 的业务建模要素
- **差距**：VBO 案例证明，需求中明确实体、字段、状态机、权限、邮件通知、菜单注册、测试数据后，AI 后续生成质量显著提高。但当前模板没有引导 BA 提供这些结构化输入。
- **建议**：在 `templates/Scenario.md` 中新增 `## Business Model Input` 节（可选，复杂场景触发）：

  ```markdown
  ## Business Model Input

  > 以下节由 BA 或需求方填写。复杂业务场景（涉及状态机、权限、多实体关联）时必填。

  ### Entities & Fields
  | Entity | Key Fields | Notes |
  |---|---|---|

  ### State Machine (if applicable)
  | From State | Event | To State | Guard Condition |
  |---|---|---|---|

  ### Permission Matrix (if applicable)
  | Role | Create | Read | Update | Delete | Special |
  |---|---|---|---|---|---|

  ### Notification Rules (if applicable)
  | Trigger | Channel | Recipient | Template |
  |---|---|---|---|

  ### Test Data (provided by BA)
  | Scenario | Input Data | Expected Result |
  |---|---|---|
  ```
- **落地文件**：`templates/Scenario.md`、`skills/scenario-analysis/SKILL.md`

### Item 28 — Solution.md 补充业务前置要素
- **状态**：✅ 已关闭（与 Item 27 重复）
- **优先级**：中
- **来源**：会议 Section 二.4 — 需求中状态机、权限、通知等前置要素决定 AI 输出质量；V4.2 Technical Constraints 只覆盖技术层面
- **现状**：`templates/Solution.md` Technical Constraints 覆盖 Java 版本、字段名、Mockito、Worktree 等技术约束，但不覆盖状态机设计、权限方案、通知方案等业务方案
- **建议**：考虑在 `templates/Solution.md` 的 Recommended Solution 节下引导填写业务方案要素：

  ```markdown
  ## Recommended Solution

  ### Business Logic Design (if applicable)
  - State Machine:
  - Permission Model:
  - Notification Strategy:
  - Menu/UI Registration:
  ```

  或作为 `## Business Design` 独立节。需评估是否与 Scenario.md Business Model Input 重复。
- **落地文件**：`templates/Solution.md`

## 三、资产库与跨案例视图

### Item 29 — 可复用资产索引
- **状态**：✅ 已完成
- **优先级**：中
- **来源**：会议 Section 五.4 — 已出现大量可复用资产（模板、Mapping 样例、状态机样例、API 测试样例、Token 优化案例、公共工具链说明），建议建立团队级资产库

- **落地文件**：`reports/Reusable_Asset_Registry.md`

### Item 30 — 跨案例 Token/ROI 汇总视图
- **状态**：✅ 已关闭（案例数不足，单案例 Cost Efficiency 节 + report_analyze_result.md 手工汇总已满足当前需求；案例积累 10+ 时重新评估）
- **优先级**：中
- **来源**：会议 Section 六后续建议 — "建立 Token / ROI 看板"；report_analyze_result.md 有多案例横向对比表但非标准化
- **现状**：每个案例有独立的 Token Report 和 Cost Efficiency，但无跨案例汇总。`report_analyze_result.md` 的多案例对比表是一次性分析产物
- **建议**：创建 `reports/Token_ROI_Dashboard.md` 模板，每次新案例完成后追加一行：

  ```markdown
  # Token / ROI Dashboard

  | Case ID | Type | Mode | Total Token | Cost | Saved Hours | Cost/Hour | Retry | Model | Result |
  |---|---|---|---:|---:|---:|---:|---:|---|---|
  ```

  配合周报 (GOVERNANCE Section 4 Weekly Review) 使用，作为团队级成本基线。
- **落地文件**：`reports/Token_ROI_Dashboard.md`
